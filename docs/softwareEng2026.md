ソフトウェア工学1回
ソフトウェア工学は、実務から体系化された知見であるため、開発経験がないとイメージしにくい。また、知識を理解しても実践することが難しく、個人だけでなくチーム全体で共有する必要がある。
また、AIによるソフトウェア開発では、要件定義・バージョン管理・テストが不可欠であり、監督者としての能力、すなわちソフトウェア工学が求められるとされている。

ソフトウェア工学２回
ソフトウェアは、情報を扱うプロダクトそのもの、またはプロダクトを提供する手段として説明されている。定義としては、命令語群、情報を扱うためのデータ構造、操作や使用法を記述した情報が含まれる。ハードウェアと異なり劣化はしない。
ソフトウェアは、新しい環境・技術・ビジネス要求に適応し、他システムと相互運用できるよう拡張・再構築される必要がある。ソフトウェア工学は、開発・運用・保守に対する体系的で規律ある、定量化可能なアプローチである。
ソフトウェア工学は、「品質」「コスト」「納期」の最適なバランスを実現するための手法・方法論である。必要性として、大規模化と複雑さ、IT人材不足、社会性と継続性、不確実性の増大が挙げられている。
また、ソフトウェア工学とプログラミングの違いとして、時間・スケール・持続可能性が重視される。保守管理では、バージョン更改やコード管理に加え、トラブル、ユーザー需要、新技術、チーム管理への対応も必要になる。

ソフトウェア工学3回
ソフトウェアライフサイクルについて、ソフトウェアはニーズの発生から始まり、企画・計画、要件定義、開発、実稼働、保守運用を経て、サービス終了や新規ソフトウェアへのリニューアルによって廃止される。
要件定義では、システムの目的・概要・機能・構成・目標性能・他システムとのインターフェイス・運用上の注意・制限事項・拡張性・開発スケジュール・開発体制・納品物などを整理する。また、要件定義では、実現する機能と実現しない機能を明確にすることが重要であり、実現性評価を通して機能の絞り込み、スケジュールの見直し、代替策の検討につなげる。
設計では、要件定義書をもとに、コーディングできるレベルの仕様書へ落とし込む。制作では、設計書に従ってシステムを構築し、内製や外部委託などの方法がある。
テストやデバッグでは、バグのないソフトウェアはないと考え、早期に発見し対処することが理想とされる。ただし、すべてをテストすることは工数の面で不可能なため、どこをどの程度テストするかを考える必要がある。
運用・保守では、稼働後のデータ活用、ソフトウェア以外の要因、開発者がプロジェクトから離れている可能性を考慮する必要がある。さらに、複数人での開発では文書による非同期コミュニケーションが重要である。

ソフトウェア工学4回
プロジェクトは、必ず終わりがある有期性と、独自の目的を達成する独自性を持つ仕事であり、ソフトウェア開発もその例である。
プロジェクトでは、現在地から進むフォアキャスティングではなく、目標から逆算して必要なことを計画するバックキャスティングが重要とされる。目標は具体的で達成する価値のあるものにし、必要に応じて修正できる。
ソフトウェア分析では、物量、実行性能、開発工数、品質などを見える化し、QCD、すなわち品質・コスト・納期の判断に使う。QCDの優先度はシステムによって異なり、人命に関わるものは品質、公共関連は費用、イベント関連は納期が重視される。
ソフトウェアの評価では、コード行数やオブジェクト容量、バグ発生率、ファンクションポイント法、使い勝手などが扱われる。バグがないことを完全に証明するのは難しく、計画したテストで対応した、またはまだ確認されていないという考え方になる。
開発プロセスには、ウォーターフォール型、スパイラルモデル、反復型、アジャイルなど複数ある。ウォーターフォール型は進捗管理がしやすく成果物が明確だが、後工程にしわ寄せが集中するリスクがある。スパイラルモデルは小さなフェーズごとにプロトタイプを作り、フィードバックを得る。アジャイルは、変化に対応し、無駄を減らし、動くソフトウェアの提供を優先する。プロセスやツールよりも個人と相互作用、ドキュメントよりも動くソフトウェア、契約よりも顧客との協力、計画よりも変化への対応を重視する。
どの開発手法がよいかは一長一短であり、場面や要求、プロジェクトマネジメントスキルによって変わる。ハイブリッド型も多く、リスクを事前に把握するリスク駆動型の考え方も示された。

ソフトウェア工学5.6回
WBSについて学んだ。WBSとは、プロジェクト目標を達成し、必要な成果物を作るために、プロジェクトチームが行う作業を階層的に分解したものである。プロジェクトを細かな作業に分解し、作業順に並べることで、スコープ、やるべき作業、全体管理、作業計画が明確になる。
WBSを作るときは、まずスコープを明確にし、トップダウンかつバックキャスティングで考える。次に大きな作業をグルーピングし、それらの相互関係を考え、各グループの作業を漏れなく重複なく洗い出す。
演習では、情報工学科学生が参加するゲーム大会を3人で企画・運営する想定でWBSを作成した。



また、Gitの基本操作について学んだ。Learning git branchingを用いて、コミット、ブランチ、HEADの分離、cherry-pick、push、pullなどを練習した。また、git init、git status、git log、git diff、git add、git commit、git reset、git revert、git clone、git pull、git push、git branch、git checkout、git mergeなど、Gitで開発を進めるための基本的なコマンドを学んだ。
第11回では、GitHubを用いたソフトウェア開発について学んだ。GitHubはリモートレポジトリのホスティングサービスであり、オープンソースソフトウェアの主要なポータルサイトである。また、GitHubフロー、clone、fork、template、issues、projectsなど、GitHub上で開発を進めるための機能について学んだ。さらに、Markdownは簡単に文章を記述する記法であり、GitHubなど多くのアプリケーションで使われていることを学んだ。
第12回では、CI/CDについて学んだ。CIはコード変更を共有リポジトリに頻繁に統合し、自動テストやビルドによってバグを早期に発見・修正するための仕組みである。CDはコード変更をテスト環境や本番環境に自動的にデプロイする仕組みである。また、CI/CDパイプラインは、ソース、ビルド、テスト、デプロイ、検証、モニタリングの流れで進むことを学んだ。GitHub Actionsでは、YAMLファイルでワークフローを定義し、トリガー、ジョブ、ステップ、アクション、ランナーを用いて自動処理を実行できることを学んだ。また演習では、GitHub Actionsを使ってflake8を実行する方法を学んだ。YAMLはインデントで階層構造を表し、キーと値のペアやリスト形式で設定を書くことを学んだ。
第13回では、GitHub Pagesを使ったCI/CDの実践について学んだ。GitHub Pagesでは、GitHub上のリモートレポジトリの内容をWebページとして公開できる。また、Jekyllを使うことでMarkdownファイルを自動でHTML化できるため、開発者はMarkdownファイルを作成すればよいことを学んだ。さらに、GitHub PagesのSourceをGitHub Actionsに設定し、Jekyllサイトを作成して、index.markdownやsoftwareEng2026.mdを編集し、プルリクエストとマージを通してWebページを公開する流れを学んだ。
